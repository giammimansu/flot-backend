v4 — Elastic & Predictive

01 Overview del progetto

Flot è un servizio di taxi-pooling da aeroporti: mette in contatto persone che atterrano nello stesso aeroporto e vanno nella stessa direzione, così possono dividere un taxi. Flot non è un taxi — è un servizio di matching. We don't drive you.

Cos'è Flot

Il prodotto risolve un problema concreto: il costo fisso elevato di un taxi dall'aeroporto al centro città. Invece di prenotare una corsa, l'utente trova qualcuno con cui condividere il taxi e ne divide il costo. Il modello di ricavo è un Trip Pass da €1,99 che ciascun utente paga per sbloccare il contatto reciproco una volta trovato un match.

⌖ Aeroporto MVP

Il lancio è su un singolo aeroporto: Milano Malpensa (MXP). L'intera architettura è però multi-aeroporto: aggiungere uno scalo significa aggiungere una entry in airports.py, senza modifiche al codice.

03 Algoritmo di matching

Il cuore del prodotto. Ogni coppia di trip candidati supera prima dei filtri binari, poi riceve uno score pesato. Le coppie sopra la soglia entrano nella compatibility matrix e vengono assegnate greedy. La v4 rende soglia e corridoio elastici nel tempo.

Formula dello score

Lo score finale combina tre componenti. In modalità Live i pesi sono 0.5 / 0.3 / 0.2; in modalità Scheduled la distanza pesa di più (0.6 / 0.2 / 0.2).

formula
score = 0.5 × distance_score
      + 0.3 × time_score
      + 0.2 × profile_score

Filtri binari pre-score

Prima di calcolare lo score, ogni coppia deve superare tutti questi check; se ne fallisce uno, la coppia è scartata.

  • Stesso aeroporto — garantito dalla struttura delle query GSI1 (un trip MXP non matcha mai con FCO).
  • Stessa direzione — TO_MILAN matcha solo con TO_MILAN.
  • GPS presenti — un trip senza destLat/destLng viene scartato silenziosamente.

Soglia per aeroporto

Ogni aeroporto definisce la sua match_threshold in airports.py. Per MXP la base è 0.25. Mai hardcodare la soglia fuori dal config.

Soglia dinamica v4 (time-decay)

La soglia non è più costante: compute_dynamic_threshold la abbassa man mano che il volo si avvicina. Lontano dal volo il sistema è selettivo (aspetta combinazioni migliori); vicino alla lock window accetta tutto ciò che è compatibile.

Ore al volo più imminenteSoglia

python · src/lib/matching.py
def compute_dynamic_threshold(
    base_threshold: float,
    flight_time_a: str,
    flight_time_b: str,
    now: datetime,
) -> float:
    """Time-decay: selettivo lontano dal volo, lasco vicino al lock."""
    dt_a = datetime.fromisoformat(flight_time_a.replace("Z", "+00:00"))
    dt_b = datetime.fromisoformat(flight_time_b.replace("Z", "+00:00"))
    hours_to_flight = min(
        (dt_a - now).total_seconds() / 3600,
        (dt_b - now).total_seconds() / 3600,
    )
    hours_to_flight = max(0, hours_to_flight)

    if hours_to_flight >= 168:  return 0.70
    if hours_to_flight >= 48:   return 0.50
    if hours_to_flight >= 24:   return 0.35
    if hours_to_flight >= 6:    return base_threshold
    return max(0.20, base_threshold - 0.05)

Corridoio direzionale v4 (detour penalty)

Al posto del limite radiale di 20 km, il sistema stima la deviazione in minuti che il tassista deve fare per servire entrambe le destinazioni nell'ordine ottimale (estimate_detour_minutes). Le rotte a "V" — destinazioni in direzioni opposte rispetto all'aeroporto — vengono penalizzate o escluse. La logica MVP usa un'approssimazione Haversine senza routing API; in v4.1 verrà sostituita con Google Routes API.

Shadow Pool v4

Il Matchmaker non crea più match definitivi immediatamente. Crea TentativeMatch silenziosi — nessuna notifica all'utente — che vengono promossi a match definitivi solo nella lock window a T‑3h. Questo permette di rivedere le assegnazioni se nel frattempo arriva una combinazione migliore. L'idempotenza è garantita da transact_write con ConditionExpression sullo stato: se il trip non è più nello stato atteso, la transazione viene annullata (TransactionCanceledException) e l'operazione saltata senza errori.

⚠ Idempotenza obbligatoria

Il Matchmaker gira ogni 5 minuti e può vedere lo stesso pool più volte. Ogni promozione a definitivo usa una condition expression (status IN (scheduled, tentative_match)); una doppia esecuzione produce un match_lock_skipped_idempotent in log, non un doppio match.

python · assegnazione greedy
def find_optimal_assignments(pairs: list[tuple]) -> list[tuple]:
    """Greedy: prende la coppia con score più alto, esclude i trip già
    accoppiati. O(n²) — sufficiente per pool MVP (<100 trip)."""
    assigned = set()
    assignments = []
    for a_id, b_id, score, dist_km, detour_min in pairs:
        if a_id in assigned or b_id in assigned:
            continue
        assignments.append((a_id, b_id, score, dist_km, detour_min))
        assigned.add(a_id); assigned.add(b_id)
    return assignments

04 Trip lifecycle

Un trip nasce scheduled (modalità primaria) o searching (Live). Attraversa lo shadow pool, il lock, il flusso di unlock e termina in uno stato terminale. Dopo il lock a T‑3h il Matchmaker non ricalcola più il trip.

Diagramma stati

scheduled→ tentative_match→ matched→ partially_unlocked→ unlocked→ completed

Stati terminali alternativi:

expired unlock_expired dissolved

Scheduled mode vs Live mode

lifecycle
SCHEDULED MODE (primaria):
  User crea trip con flightTime futuro → status: "scheduled"
    → Matchmaker (ogni 5 min) cerca candidati ±60 min
    → tentative match silenzioso → lock a T-3h → "matched"
    → notifica Push + Email + WS
    → nessun match entro flightTime + 2h → "expired"

LIVE MODE (secondaria):
  User crea trip con flightTime = now → status: "searching"
    → matching sincrono nell'handler search
    → match → "matched" → notifica WS
    → timeout 5 min → "expired"

Scheduled Rule EventBridge (ogni 5 minuti)

Una EventBridge Scheduled Rule invoca il MatchmakerFunction ogni 5 minuti. Il job legge via GSI5 tutti i trip scheduled di un aeroporto, costruisce la compatibility matrix con soglia dinamica e filtro detour, e aggiorna lo shadow pool.

Lock window T‑3h

⌖ Regola del lock

A T‑3h dal volo i tentative match vengono promossi a definitivi. Dopo il lock il Matchmaker non ricalcola più quel trip: la coppia è congelata e i due utenti vengono notificati. Questo dà certezza all'utente vicino alla partenza.

Flight Tracker v4

Quando un trip include flightNumber (obbligatorio in v4), il FlightTrackerFunction interroga periodicamente l'API di tracking voli per ottenere l'ETA reale e aggiornare automaticamente flightTime. Se un volo è in ritardo, l'evento on_flight_delayed verifica se il match esistente è ancora compatibile.

⚠ Circuit breaker + cascade failover

Ogni provider ha un circuit breaker indipendente (3 fail → 30 min blackout). La strategia a cascata è:

  1. Provider primario (FLIGHT_TRACKER_PROVIDER)
  2. Provider fallback (FLIGHT_TRACKER_FALLBACK_PROVIDER) — se configurato
  3. Degraded — entrambi falliti: fetch_flight_eta ritorna None, il trip resta nel pool con trackingStatus="degraded" e usa il flightTime statico. Non viene escluso dal matching.

Quando un breaker si apre, viene emessa la metrica CloudWatch FlightTrackerCircuitOpen (alarm-ready).

05 Payment Deadlock Resolution

Un match richiede che entrambi gli utenti paghino €1,99 per sbloccare il contatto. Il deadlock nasce quando solo un utente sblocca. La soluzione Smart Auto-Capture elimina il rischio percepito senza aggiungere step al funnel.

Il problema

  • Gioco del pollo — nessuno vuole pagare per primo ("e se l'altro non paga?").
  • Asimmetria temporale — A paga, B è in volo / distratto / ignora la notifica.
  • Trust debt — A vede l'auth pendente sull'estratto conto e si sente truffato.

La soluzione: Smart Auto-Capture

Il funnel resta identico alla v3, zero step aggiuntivi. Le differenze:

  • Copy di trust sotto il CTA: "Charged ONLY when both unlock. No mutual unlock = no charge."
  • Stato intermedio partially_unlocked: quando un utente sblocca, il partner riceve notifiche urgenti con pressione sociale ("Marco ha già sbloccato!").
  • Timeout con auto-void: se il partner non risponde entro unlock_timeout_minutes (MXP: 120 min), l'auth del primo viene annullata e il trip torna nel pool.
  • Re-pool: il Matchmaker cerca automaticamente un nuovo partner per il trip ri-immesso.

Flusso

flow
Utente A preme "Unlock"
  → Stripe PaymentIntent (capture_method: manual) → auth hold €1.99
  → Match status: "partially_unlocked"
  → unlockDeadline = firstUnlockAt + unlock_timeout_minutes
  → emette: match.partially_unlocked
  → notifica URGENTE a B: "[A] ha sbloccato!"
  → crea EventBridge Scheduler one-shot per il timeout

Utente B sblocca entro la deadline
  → CAPTURE SIMULTANEO dei due PaymentIntent
  → Match status: "unlocked" → chat abilitata

B NON risponde entro la deadline
  → void PaymentIntent A (€0 addebitati)
  → Match status: "unlock_expired"
  → re-pool di entrambi i trip
  → previousMatchPartners += partner non responsivo
✖ Mai capture senza entrambi gli auth — ordine di sicurezza

Il capture avviene in questo ordine preciso per garantire zero charge unilaterali:

  1. Capture secondo PI (utente B) — se fallisce, viene cancellato; il primo PI di A non viene mai toccato → €0 addebitati.
  2. Capture primo PI (utente A) — se fallisce dopo che il secondo è già stato catturato, viene emesso un Refund immediato sul secondo → €0 addebitati.

Questo ordine (secondo prima del primo) era invertito nelle versioni precedenti, esponendo al rischio di addebitare B senza addebitare A.

previousMatchPartners

Per evitare loop infiniti, un trip re-pooled porta la lista previousMatchPartners. Il Matchmaker filtra i candidati escludendo i partner non responsivi già incontrati, così lo stesso match fallito non si ripete.

ⓘ Note di implementazione

partially_unlocked è uno stato del Match, non del Trip: il trip resta matched durante l'attesa. I reminder sono EventBridge Scheduler one-shot con ActionAfterCompletion: DELETE, non polling. Il timeout è per-aeroporto (unlock_timeout_minutes), mai hardcoded. Con FAKE_DOOR_MODE=true l'unlock registra l'intent senza creare il PaymentIntent, ma simula comunque il flusso per testare la UX.

#3 — State Machine esplicita (src/lib/state_machine.py)

Ogni aggiornamento di status su Trip e Match passa ora attraverso TripStateMachine.transition(from, to) o MatchStateMachine.transition(from, to). Se la transizione non è nella lista degli archi ammessi, viene sollevata InvalidTransitionError.

#4 — Race condition Matchmaker (bug fix)

Trovata e corretta una race condition in dynamo.create_tentative_match: il ConditionExpression sui trip era attribute_exists(pk) — verificava solo l'esistenza, non lo stato. Due run concorrenti potevano entrambi leggere i trip come scheduled e creare TentativeMatch duplicati per la stessa coppia.

Fix: condizione cambiata in #status = :scheduled. Il run più lento riceve TransactionCanceledException e skippa (tentative_match_concurrent_skip nel log). optimize_pool gestisce l'eccezione con continue.

ⓘ Copertura test concorrenza

4 test in tests/integration/test_matchmaker_concurrency.py usano threading + moto per verificare: nessun trip matchato due volte in run paralleli, al massimo 1 lock definitivo da run concorrenti, pool dispari senza doppi assignment, dissolve durante promozione lock = skip senza crash.

08 Regole di sviluppo non negoziabili

Queste regole sono inviolabili. Se un task richiede di romperne una, ci si ferma e lo si segnala prima di scrivere codice.

  • 01 Non hardcodare mai dati aeroporto fuori da airports.py / airportStore.
  • 02 Ogni Trip e ogni Match devono portare airportCode.
  • 03 Il blur della foto partner è sempre server-side (Gaussian σ=15).
  • 04 Il Matchmaker deve essere idempotente (transact_write + condition expression).
  • 05 Trip senza destLat/destLng sono esclusi silenziosamente dal matching.
  • 06 Mai print() nel backend — solo logger.info() di Lambda Powertools.
  • 07 Float in log e response: arrotondati a 2–3 decimali.
  • 08 FAKE_DOOR_MODE=true va controllato prima di ogni capture Stripe.
  • 09 capture_method: 'manual' obbligatorio per tutti i PaymentIntent.
  • 10 TTL obbligatorio: chat messages 48h, WebSocket connections 24h.
  • 11 Multi-entity updates DynamoDB: sempre transact_write_items.
  • 12 TentativeMatch è silenzioso: nessuna notifica utente fino a match.locked.
